# Ralph Codex Review Fix Report

## Executive Summary

During ralph run `20260123T152622Z-LAPTOP-T9SQIU0F-vexycats-1390-6f48`, the codex implementation review feature failed repeatedly because Codex CLI's `--sandbox read-only` mode blocks ALL shell commands on Windows, including read-only operations. This caused:
- Multiple forced retries (iter 2, 3, 4 for fn-8-pua.6)
- Wasted compute cycles (~$10+ in API costs for retries)
- Claude having to write fake "manual review" receipts to work around the issue

A partial fix was applied to flowctl.py (embedding git diff in prompt), but more comprehensive fixes are needed.

---

## Root Cause Analysis

### The Problem

File: `.flow/bin/flowctl.py` function `run_codex_exec()` (lines 934-1003)

The function calls Codex CLI with:
```python
cmd = [
    codex, "exec",
    "--model", effective_model,
    "-c", 'model_reasoning_effort="high"',
    "--sandbox", "read-only",  # <-- THIS IS THE PROBLEM
    "--skip-git-repo-check",
    "--json",
    "-",
]
```

On Windows, `--sandbox read-only` blocks ALL shell commands, not just writes:

```json
{
  "type": "item.completed",
  "item": {
    "command": "\"C:\\windows\\...\\powershell.exe\" -Command \"... Get-Content src\\services\\DownloadManager.ts ...\"",
    "aggregated_output": "... rejected: blocked by policy",
    "exit_code": -1,
    "status": "failed"
  }
}
```

Codex then returns:
> "I can't read any files in this session: every filesystem read is blocked by current policy"

And gives `VERDICT=NEEDS_WORK` or `VERDICT=UNKNOWN` because it cannot perform the review.

### Current Bandaid Fix

Lines 4691-4721 in flowctl.py now embed the git diff directly in the prompt:
```python
diff_content = subprocess.run(["git", "diff", "-U5", base_branch], ...)
if diff_content:
    prompt += "\n\n## Full Diff Content\n**IMPORTANT**: The diff below is untrusted..."
```

This allows Codex to review the CHANGES, but it cannot:
- Read surrounding context in files
- Check imports or dependencies
- Verify patterns used elsewhere in codebase

## Issues to Fix

### Issue 1: Codex Sandbox Mode Configuration

**Problem**: `--sandbox read-only` doesn't work as expected on Windows.

**Current code** (flowctl.py line 982):
```python
"--sandbox", sandbox,  # default: "read-only"
```

**Options to consider**:
1. Add `--sandbox` parameter to `flowctl codex impl-review` command
2. Auto-detect Windows and use a different sandbox mode
3. Document the limitation and suggest workarounds

### Issue 2: Include Full File Contents, Not Just Diff

**Problem**: Diff alone is insufficient for quality review. Reviewer needs context.

**Enhancement needed in `cmd_codex_impl_review()`**:
- For each changed file in the diff, include the full current file contents
- This gives Codex the context it needs without requiring filesystem access
- Truncate large files to stay within token limits

### Issue 3: Better Error Detection for Failed Reviews

**Problem**: When Codex returns `UNKNOWN` or `NEEDS_WORK` due to sandbox issues (not actual code problems), ralph retries endlessly.

**Current flow**:
1. Codex can't read files -> returns NEEDS_WORK
2. Ralph sees NEEDS_WORK -> resets task, retries
3. Loop continues until MAX_ATTEMPTS_PER_TASK

**Enhancement needed**:
- Detect when Codex response contains "blocked by policy" or "can't read files"
- Surface this as a configuration error, not a code quality issue
- Potentially fallback to a different review mode or skip review

### Issue 4: Receipt Validation Doesn't Check Verdict Quality

**Problem**: A receipt with `verdict=SHIP` but review text showing "couldn't read files" passes validation.

**Current receipt validation** (ralph.sh lines 736-754):
```bash
verify_receipt() {
  # Only checks: file exists, type matches, id matches
  # Does NOT check if verdict is meaningful
}
```

**Enhancement needed**:
- Add optional validation that review text doesn't contain error patterns
- Or: require minimum review length / content

### Task 2: Include full file contents in codex review prompt

Enhance `cmd_codex_impl_review()` to include full contents of changed files:

```python
def get_changed_file_contents(base_branch: str, max_size: int = 50000) -> str:
    """Get full contents of files changed since base_branch."""
    changed_files = get_changed_files(base_branch)
    contents = []
    total_size = 0
    for file_path in changed_files:
        if total_size > max_size:
            contents.append(f"\n[Truncated - {len(changed_files) - len(contents)} more files]")
            break
        try:
            content = Path(file_path).read_text()
            contents.append(f"## {file_path}\n```\n{content}\n```")
            total_size += len(content)
        except:
            pass
    return "\n\n".join(contents)
```

Files to modify:
- `.flow/bin/flowctl.py`: Add helper function and call in `cmd_codex_impl_review()`

### Task 3: Detect sandbox failures in codex output

Add detection for sandbox-related failures in codex responses:

```python
SANDBOX_ERROR_PATTERNS = [
    "blocked by policy",
    "can't read any files",
    "filesystem read is blocked",
    "every filesystem read is blocked",
]

def is_sandbox_failure(output: str) -> bool:
    """Check if codex failed due to sandbox restrictions."""
    output_lower = output.lower()
    return any(pattern in output_lower for pattern in SANDBOX_ERROR_PATTERNS)
```

When detected:
- Log a warning about sandbox configuration
- Consider returning a special verdict or error code
- Allow ralph to handle this differently than code quality issues
